## 一、服务器选型建议

### 1.1 云服务商选择

| 服务商 | 推荐场景 | 优势 | 劣势 |
|--------|----------|------|------|
| **阿里云** | 首选 | 国内CDN覆盖好，ECS稳定，适合游戏类应用 | 价格中等 |
| **腾讯云** | 备选 | 与微信QQ登录生态契合，费用较低 | 国际化支持一般 |
| **华为云** | 备选 | 政企客户友好，性能稳定 | 生态相对较小 |

**推荐配置：初期使用阿里云，后续可扩展至多云架构**

### 1.2 服务器规格建议

#### 开发/测试环境
- **规格**：2核4G × 2台（负载均衡）
- **系统**：CentOS 7.9 / Ubuntu 22.04 LTS
- **带宽**：5Mbps
- **存储**：100GB SSD

#### 生产环境（预估）

| 阶段 | CPU | 内存 | 带宽 | 存储 | 并发预估 |
|------|-----|------|------|------|----------|
| MVP初期 | 4核 | 8GB | 10Mbps | 200GB SSD | 500 |
| 成长期 | 8核 | 16GB | 50Mbps | 500GB SSD | 2000 |
| 成熟期 | 16核 | 32GB | 100Mbps | 1TB SSD | 10000+ |

### 1.3 扩展策略
- **垂直扩展**：先升级单机配置
- **水平扩展**：增加Web服务器，配合负载均衡
- **预留弹性**：使用弹性伸缩（ESS）应对突发流量

## 三、CI/CD流程设计

### 3.1 流水线阶段

```
代码提交 → Git Hooks → 构建镜像 → 单元测试 → 镜像推送 
    ↓
集成测试 (Docker Compose) → 镜像签名 → 灰度发布 → 全量发布
    ↓
通知 (钉钉/企微) → 监控确认
```

### 3.2 GitLab CI 配置示例

```yaml
# .gitlab-ci.yml
stages:
  - build
  - test
  - deploy

variables:
  DOCKER_REGISTRY: registry.cn-hangzhou.aliyuncs.com
  APP_NAME: lok-world-api

.build:
  stage: build
  image: docker:24.0
  services:
    - docker:24.0-dind
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $DOCKER_REGISTRY
    - docker build -t $DOCKER_REGISTRY/$APP_NAME:$CI_COMMIT_SHA .
    - docker push $DOCKER_REGISTRY/$APP_NAME:$CI_COMMIT_SHA
  only:
    - main
    - develop

.test:
  stage: test
  image: python:3.11
  services:
    - mysql:8.0
    - redis:7.0
  script:
    - pip install -r requirements.txt
    - pytest tests/ --junitxml=report.xml
  coverage: '/TOTAL.*\s+(\d+%)$/'
  artifacts:
    reports:
      junit: report.xml

.deploy-staging:
  stage: deploy
  script:
    - kubectl set image deployment/$APP_NAME app=$DOCKER_REGISTRY/$APP_NAME:$CI_COMMIT_SHA
  environment:
    name: staging
  only:
    - develop

.deploy-production:
  stage: deploy
  script:
    - kubectl set image deployment/$APP_NAME app=$DOCKER_REGISTRY/$APP_NAME:$CI_COMMIT_SHA
    - kubectl rollout status deployment/$APP_NAME
  environment:
    name: production
  when: manual
  only:
    - main
```

```
│                   (仪表盘 / 告警规则 / 报表)                   │

                    ▲                    ▲
                    │                    │
        ┌───────────┴───────────┐ ┌───────┴────────┐
        │   Prometheus Server   │ │ AlertManager   │
        │    (指标采集/存储)      │ │   (告警触发)    │
        └───────────┬───────────┘ └───────┬────────┘
                    │                     │
    ┌───────────────┼─────────────────────┼───────────────┐
    │               │                     │               │
┌───┴───┐      ┌────┴────┐         ┌──────┴─────┐   ┌──────┴─────┐
│Node   │      │Docker   │         │Application │   │  日志采集   │

    │               │                     │               │
    └───────────────┴─────────────────────┴───────────────┘
                        被监控对象
```

### 4.2 监控指标体系

| 类别 | 指标 | 告警阈值 | 采集方式 |
|------|------|----------|----------|
| **服务器** | CPU使用率 | >80%持续5min | Node Exporter |
| | 内存使用率 | >85% | Node Exporter |
| | 磁盘使用率 | >90% | Node Exporter |
| | 网络流量 | 带宽>80% | Node Exporter |
| **Docker** | 容器状态 | Down立即告警 | cAdvisor |
| | 资源限制 | 内存>90% | cAdvisor |
| **应用** | API响应时间 | P99>2s | APM |
| | 错误率 | >1% | 自定义埋点 |
| | QPS | 突增200% | Nginx metrics |
| **数据库** | 连接数 | >80%max | MySQL Exporter |
| | 慢查询 | >100ms | MySQL日志 |
| | 主从延迟 | >5s | MySQL Exporter |

### 4.3 告警渠道

| 级别 | 触发条件 | 通知方式 | 响应时效 |
|------|----------|----------|----------|
| P0-紧急 | 服务不可用、数据库Down | 电话+短信+企微 | 5分钟内 |
| P1-严重 | 错误率>5%、响应超时 | 企微机器人 | 15分钟内 |
| P2-警告 | CPU>80%、告警阈值逼近 | 企微消息 | 1小时内 |
| P3-提示 | 例行检查、容量预警 | 邮件 | 工作时间 |

### 4.4 关键告警规则示例 (Prometheus)

```yaml
# alerting_rules.yml
groups:
  - name: lok-world-alerts
    rules:
      - alert: APIHighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.01
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "API错误率超过1%"

      - alert: ServiceDown
        expr: up == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "服务 {{ $labels.instance }} 不可用"

      - alert: HighMemoryUsage
        expr: (node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes) / node_memory_MemTotal_bytes > 0.85
        for: 5m
        labels:
          severity: warning
```

## 五、备份与容灾方案

### 5.1 备份策略

| 数据类型 | 备份频率 | 保留周期 | 存储位置 |
|----------|----------|----------|----------|
| 数据库全量 | 每天凌晨2:00 | 30天 | OSS跨区域 |
| 数据库增量 | 每小时 | 7天 | 本地+OSS |
| 配置文件 | 变更时 | 90天 | OSS |
| 用户上传文件 | 实时同步 | - | OSS多副本 |
| 代码仓库 | 自动 | 永久 | GitLab + 镜像 |

### 5.2 备份脚本示例

```bash
#!/bin/bash
# backup.sh - 数据库备份脚本

set -e

# 配置
DB_HOST="localhost"
DB_NAME="lokworld"
DB_USER="backup"
DB_PASS="xxx"
OSS_BUCKET="lok-world-backup"
BACKUP_DIR="/data/backup"
DATE=$(date +%Y%m%d_%H%M%S)

# 创建备份目录
mkdir -p $BACKUP_DIR

# 全量备份
echo "[$(date)] Starting full backup..."
mysqldump -h$DB_HOST -u$DB_USER -p$DB_PASS --single-transaction \
  --routines --triggers --events $DB_NAME | gzip > $BACKUP_DIR/${DB_NAME}_${DATE}.sql.gz

# 校验备份
if [ $? -eq 0 ]; then
    echo "[$(date)] Backup completed successfully"
    # 上传到OSS
    ossutil cp $BACKUP_DIR/${DB_NAME}_${DATE}.sql.gz oss://$OSS_BUCKET/
    # 清理本地超过30天的备份
    find $BACKUP_DIR -name "*.sql.gz" -mtime +30 -delete
else
    echo "[$(date)] Backup failed!"
    exit 1
fi
```

### 5.3 容灾方案

#### 灾难恢复等级 (RTO/RPO)

| 等级 | RTO | RPO | 方案 | 成本 |
|------|-----|-----|------|------|
| 基础 | 4小时 | 1天 | 每日备份 | 低 |
| 标准 | 1小时 | 1小时 | 实时备份+从库 | 中 |
| 高级 | 15分钟 | 5分钟 | 主从+跨区域 | 高 |

**推荐：MVP阶段采用"基础+标准混合"方案**

#### 跨区域容灾架构
```
主站点 (华东1) ───── 同步复制 ──────→ 备站点 (华北2)
     │                                        │
     └───────── 异步备份 (OSS跨区域) ──────────┘
```

### 5.4 恢复演练

## 六、安全加固清单

### 6.1 服务器安全

- [ ] **SSH安全**
  - 禁用root远程登录
  - 使用密钥登录，禁止密码
  - 修改默认端口22 → 22022
  - 配置 fail2ban 防暴力破解

- [ ] **系统加固**
  - 关闭不必要的服务/端口
  - 启用防火墙 (iptables/firewalld)
  - 限制用户sudo权限
  - 定期更新系统补丁 (yum-cron/unattended-upgrades)

- [ ] **基线检查**
  - 部署 OSSEC 主机入侵检测
  - 配置 Auditd 审计日志

### 6.2 应用安全

- [ ] **API安全**
  - HTTPS全站加密 (Let's Encrypt)
  - API请求签名验证
  - 请求频率限制 (Redis计数)
  - 参数校验与SQL注入防护
  - CORS跨域配置

- [ ] **认证授权**
  - JWT Token安全存储
  - Refresh Token轮换机制
  - 敏感操作二次验证
  - 权限最小化原则

- [ ] **数据安全**
  - 敏感信息加密存储 (密码加盐哈希)
  - 日志脱敏处理
  - 定期密钥轮换

### 6.3 网络安全

- [ ] **边界防护**
  - WAF防火墙 (阿里云WAF)
  - DDoS基础防护
  - 安全组精细化配置

- [ ] **内网隔离**
  - 数据库不对外开放
  - 内网服务间认证
  - VPC网络隔离

### 7.2 日志规范

#### 日志分级

| 级别 | 标识 | 场景 | 示例 |
|------|------|------|------|
| DEBUG | debug | 开发调试 | 请求参数、变量值 |
| INFO | info | 正常运行 | 用户登录、API调用 |
| WARNING | warn | 潜在问题 | 资源逼近阈值 |
| ERROR | error | 错误异常 | DB连接失败 |
| FATAL | fatal | 严重故障 | 服务不可用 |

#### 日志格式 (JSON)

```json
{
  "timestamp": "2026-04-22T10:14:00.000Z",
  "level": "INFO",
  "service": "lok-world-api",
  "trace_id": "abc123",
  "user_id": 10001,
  "action": "user_login",
  "ip": "192.168.1.100",
  "message": "User logged in successfully",
  "duration_ms": 45,
  "metadata": {}
}
```

#### Python 日志配置示例

```python
# logging_config.py
import logging
import json
from datetime import datetime

class JSONFormatter(logging.Formatter):
    def format(self, record):
        log_data = {
            "timestamp": datetime.utcnow().isoformat(),
            "level": record.levelname,
            "service": "lok-world-api",
            "trace_id": getattr(record, 'trace_id', '-'),
            "message": record.getMessage(),
            "logger": record.name,
        }
        if record.exc_info:
            log_data["exception"] = self.formatException(record.exc_info)
        return json.dumps(log_data)

# 配置
logging.basicConfig(
    handlers=[
        logging.StreamHandler(),
        logging.FileHandler("/var/log/lokworld/app.log")
    ],
    formatter=JSONFormatter(),
    level=logging.INFO
)
```
