## 一、项目阶段划分

### 1.1 阶段总览

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           项目生命周期 (预计16周)                        │
├─────────────┬─────────────┬─────────────┬─────────────┬────────────────┤
│   规划期    │   开发期    │   测试期    │   上线期    │     运营期     │
│  (2周)     │   (10周)    │   (2周)     │   (1周)     │    (持续)      │
│ W1-W2      │   W3-W12    │   W13-W14   │   W15       │    W16+        │
└─────────────┴─────────────┴─────────────┴─────────────┴────────────────┘
```

### 1.2 各阶段详细说明

| 阶段 | 周次 | 核心目标 | 关键产出 |
|------|------|----------|----------|
| **规划期** | W1-W2 | 需求对齐、技术方案确认 | PRD、技术方案、架构设计、Sprint规划 |
| **开发期-S1** | W3-W5 | 基础设施+核心功能MVP | 用户体系、宠物查询基础、悬浮窗 |
| **开发期-S2** | W6-W8 | 论坛功能+第三方登录 | 帖子模块、评论、微信/QQ登录 |
| **开发期-S3** | W9-W10 | 后台管理+多端适配 | CMS系统、PC/Android适配 |
| **开发期-S4** | W11-W12 | 优化与集成 | 性能优化、UI优化、全流程联调 |
| **测试期** | W13-W14 | 全面测试+修复 | 测试报告、Bug修复、验收 |
| **上线期** | W15 | 灰度发布+正式上线 | 部署文档、监控告警、运营准备 |
| **运营期** | W16+ | 持续迭代 | 迭代计划、数据监控、用户反馈 |

## 二、关键里程碑

### 2.1 里程碑总览

```
M1 (W2) 规划冻结 ─────── M2 (W5) MVP完成 ─────── M3 (W8) 功能冻结 ─────── M4 (W12) 测试就绪 ─────── M5 (W15) 正式上线
   │                      │                       │                       │                       │
   ▼                      ▼                       ▼                       ▼                       ▼
 PRD定稿                登录可用                 论坛可用                 全量测试通过             APP商店上架

```

### 2.2 里程碑详情

| 里程碑 | 时间 | 验收标准 | 责任人 |
|--------|------|----------|--------|
| **M1: 规划冻结** | W2周五 | ✅ PRD签字确认 ✅ 技术方案评审通过 ✅ 架构设计定稿 ✅ Sprint计划发布 | 产品经理 + 项目经理 |
| **M2: MVP完成** | W5周五 | ✅ 用户注册/登录可用 ✅ 宠物查询基础功能 ✅ 悬浮窗核心交互 | 前端主程 + 后端主程 |
| **M3: 功能冻结** | W8周五 | ✅ 论坛全部功能 ✅ 微信/QQ登录 ✅ 消息通知 | 全体开发 |
| **M4: 测试就绪** | W12周五 | ✅ 功能无阻塞Bug ✅ 性能达标 ✅ 安全扫描通过 | 测试负责人 |
| **M5: 正式上线** | W15周五 | ✅ 灰度用户>1000 ✅ 崩溃率<0.1% ✅ 运营数据看板就绪 | 运维 + 运营 |

## 三、团队协作流程

### 3.1 会议机制

| 会议类型 | 频率 | 时间 | 时长 | 参与者 | 目的 |
|----------|------|------|------|--------|------|
| **每日站会** | 每日 | 9:30 | 15min | 全体开发 | 同步进度、识别阻塞 |
| **Sprint规划会** | 每2周 | 周一 | 2h | 产品+开发+测试 | 规划Sprint内容 |
| **Sprint评审会** | 每2周 | Sprint最后一天 | 1h | 全体 | 演示成果、收集反馈 |
| **周例会** | 每周 | 周五 | 1h | 全体 | 进度回顾、风险同步 |
| **风险评审会** | 每2周 | 周三 | 30min | 产品+开发+运维 | 评估风险状态 |

### 3.2 沟通渠道

```
┌─────────────────────────────────────────────────────────────────┐
│                        沟通矩阵                                  │
├──────────────────┬─────────────────────────────────────────────┤
│ 实时沟通          │ 微信群 / 飞书群 (按模块分组)                  │
│ 文档协作          │ 腾讯文档 / 语雀 (需求文档、设计稿)            │
│ 代码协作          │ GitLab (dev分支开发 → review → main合并)    │
│ 任务管理          │ TAPD / 禅道 (任务分配、状态跟踪)             │
│ 缺陷管理          │ TAPD缺陷单 (按优先级处理)                   │
└──────────────────┴─────────────────────────────────────────────┘
```

### 3.3 开发流程

```
需求 → 设计 → 开发 → CodeReview → 测试 → 预发布 → 正式发布
 │      │      │        │          │      │         │
 ▼      ▼      ▼        ▼          ▼      ▼         ▼
 PRD   UI/UE  编码    主程审核   冒烟+功能  灰度1%   全量
```

## 四、风险管理清单

### 4.1 风险矩阵

| # | 风险描述 | 影响 | 概率 | 应对策略 | 责任人 |
|---|----------|------|------|----------|--------|
| **R1** | 第三方登录接口申请延误（微信/QQ审核周期长） | 🔴 高 | 🟡 中 | 提前2周启动申请，预留3周缓冲；优先实现手机号登录作为备选 | 后端主程 |
| **R2** | PC端EXE打包兼容性（Windows多版本兼容） | 🔴 高 | 🟡 中 | 使用Tauri/Electron LTS版本；建立多版本测试矩阵 | 前端主程 |
| **R3** | 悬浮窗权限适配（Android多版本权限差异） | 🟡 中 | 🟡 中 | 梳理权限矩阵；分版本实现降级方案 | Android开发 |
| **R4** | 核心模块技术难点超时（宠物数据接口、论坛高并发） | 🔴 高 | 🟡 中 | W3前完成POC验证；预留20%缓冲时间 | 后端主程 |
| **R5** | 人员变动导致进度延迟 | 🟡 中 | 🟢 低 | 文档化关键知识；关键岗位AB角备份 | 项目经理 |
| **R6** | APP商店审核被拒（签名、隐私政策问题） | 🟡 中 | 🟡 中 | 提前准备隐私政策；预置合规检查清单 | 运营+测试 |
| **R7** | 需求变更打乱开发节奏 | 🟡 中 | 🟢 低 | 变更需评审；统一纳入下一Sprint | 产品经理 |

### 4.2 风险升级机制

```
Level 1: 日常风险 → 项目经理处理
Level 2: 阻塞性风险 → 项目经理升级至team-lead
Level 3: 重大危机 → team-lead协调资源解决
```

## 五、与产品经理协作机制

### 5.1 协作分工

| 事项 | 产品经理职责 | 项目经理职责 | 决策机制 |
|------|-------------|-------------|----------|
| 需求制定 | ✅ 主导PRD编写 | 参与评审、提供技术约束 | 产品决策，技 |
| 优先级排序 | 提建议 | ✅ 最终协调确认 | 术可行性由后 |
| 进度把控 | 了解进度 | ✅ 主导进度管理 | 端主程确认 |
| 需求变更 | 提出变更申请 | ✅ 管理变更流程 | 评审会决策 |
| 质量验收 | 功能验收 | ✅ 流程质量把控 | 测试主导 |

### 5.2 协作流程

```
┌─────────────────────────────────────────────────────────────────┐
│                     双周迭代协作流程                              │
│                                                                  │
│  Sprint规划(W1周一)                                               │
│       │                                                         │
│       ▼                                                         │
│  产品：提需求 → 开发：估工时 → 共同：排优先级 → 输出Sprint计划     │
│       │                                                         │
│       ▼                                                         │
│  Sprint执行(2周)                                                 │
│       │                                                         │
│       ├── 每日站会同步 ──→ 阻塞问题快速升级                       │
│       │                                                         │
│       ├── 需求答疑 ──→ 产品24h内响应                              │
│       │                                                         │
│       └── 变更评估 ──→ 评估影响 → 确认是否纳入                     │
│       │                                                         │
│       ▼                                                         │
│  Sprint评审(W2周五)                                              │
│       └── 演示 → 反馈 → 记录优化项 → 下轮迭代纳入                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.3 约定机制

| 约定事项 | 规则 |
|----------|------|
| 需求响应时间 | 产品经理需在24小时内响应开发疑问 |
| 变更响应时间 | 变更申请需在48小时内给出评估结果 |
| 验收时间 | 功能提测后5个工作日内完成验收 |
| 会议预约 | 提前1天预约，紧急会议除外 |
