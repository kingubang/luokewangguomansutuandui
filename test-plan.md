## 一、测试策略

### 1.1 测试阶段划分

| 阶段 | 时机 | 参与方 | 重点 | 持续时间 |
|------|------|--------|------|----------|
| **需求评审测试** | 需求评审阶段 | QA + 产品经理 | 需求可测试性、边界条件、异常场景 | 每个Sprint Day 1 |
| **测试用例设计** | 开发启动同步 | QA | 用例编写与评审 | 与开发并行 |
| **单元测试** | 编码阶段 | 开发 | 模块级逻辑正确性 | 贯穿开发周期 |
| **集成测试** | 模块联调后 | QA + 前后端 | 前后端接口联调、数据流转 | Sprint 中后期 |
| **系统测试** | 功能集成后 | QA | 全流程功能验证、UI、兼容性 | 每Sprint末 2-3天 |
| **验收测试(UAT)** | 系统测试通过 | QA + 产品经理 + 运营 | 业务流程验收 | 每Sprint末 1天 |
| **回归测试** | Bug修复后 | QA | 确认修复不影响现有功能 | Bug修复后即刻 |

### 1.2 测试类型矩阵

|----------|--------|---------|------|------|-----------|

| 安全测试 | P1 | 基础 | ✅ | ✅ | 工具辅助 |

### 1.3 准入准出标准

**准入标准（进入系统测试）：**
- [ ] 所有计划功能开发完成，单元测试通过率 ≥ 80%
- [ ] 前后端接口联调完成，核心接口无 P0/P1 Bug
- [ ] 测试环境搭建完成，测试数据准备就绪
- [ ] 产品经理完成需求走查，无需求歧义
- [ ] 冒烟测试用例全部通过

**准出标准（达到发布条件）：**
- [ ] P0 Bug = 0，P1 Bug = 0
- [ ] P2 Bug ≤ 3 且不影响核心流程
- [ ] 测试用例执行率 = 100%
- [ ] 用例通过率 ≥ 95%
- [ ] 兼容性测试无阻断性问题
- [ ] 性能指标满足基线要求（首屏加载 ≤ 3s，API响应 ≤ 2s）

## 二、测试用例编写计划

### 2.1 优先级排序原则

| 优先级 | 定义 | 用例占比 | 覆盖范围 |
|--------|------|----------|----------|
| **P0-冒烟** | 核心主流程，阻塞发布 | ~15% | 登录、宠物查询主流程、论坛发帖/浏览、用户中心 |
| **P1-核心** | 重要功能 + 常见异常 | ~35% | 第三方登录、搜索、帖子操作、消息通知、数据展示 |
| **P2-一般** | 边界条件 + 次要功能 | ~30% | 分页、排序、输入校验、文案检查、次要交互 |
| **P3-低** | 罕见场景 + 体验优化 | ~20% | 极端数据、冷门路径、视觉细节 |

### 2.2 用例编写排期（按Sprint）

| Sprint | 编写重点 | 用例预估数量 | 评审时间 |
|--------|----------|-------------|----------|
| Sprint 1-2 (MVP核心) | 用户注册/登录、第三方登录、宠物查询 | 120+ | Sprint 2 Day 2 |
| Sprint 3-4 (MVP论坛) | 论坛发帖/回复/点赞、用户中心 | 100+ | Sprint 4 Day 2 |
| Sprint 5-6 (v1.0增强) | 悬浮窗、后台管理、性能 | 80+ | Sprint 6 Day 2 |
| Sprint 7-8 (v1.1优化) | 全面回归 + 新增功能 | 60+ | Sprint 8 Day 2 |
| **合计** | | **360+** | |

### 2.3 用例设计方法

- **等价类划分**：输入参数覆盖（搜索关键词长度、手机号格式等）
- **边界值分析**：极值场景（帖子字数上限/下限、分页边界）
- **状态迁移**：用户状态（未登录→登录→授权→绑定→解绑）
- **场景法**：核心业务流程（注册→登录→查宠物→发帖→互动）
- **错误推测**：基于经验预判（网络中断、并发操作、特殊字符）

## 六、自动化测试方案

### 6.1 自动化分层策略

```
┌─────────────────────────────────────┐
│         E2E自动化 (15%)              │  ← 关键主流程
├─────────────────────────────────────┤
│       集成/接口自动化 (50%)           │  ← API接口全覆盖
├─────────────────────────────────────┤
│       单元测试 (35%)                 │  ← 开发自测
└─────────────────────────────────────┘
```

### 6.2 接口自动化（优先级最高）

**工具选型**：Python + pytest + requests + Allure

| 模块 | 自动化接口数 | 优先级 |
|------|-------------|--------|
| 用户模块（注册/登录/信息） | 15+ | P0 |
| 宠物查询模块 | 10+ | P0 |
| 论坛模块（帖子/回复/点赞） | 20+ | P0 |
| 第三方登录 | 8+ | P1 |
| 后台管理 | 15+ | P1 |
| **合计** | **68+** | |

**接口自动化覆盖：**
- 正向功能验证（参数正确→期望结果）
- 参数校验（缺失/错误参数→正确错误码）
- 权限验证（未登录/无权限→401/403）
- 边界值（分页极值、字符串长度极值）
- 数据一致性（创建→查询→修改→删除完整链路）

### 6.3 E2E自动化

**PC端（EXE）**：Python + pywinauto / Playwright（内嵌WebView部分）

| 自动化场景 | 描述 |
|-----------|------|
| 登录主流程 | 输入账号密码→登录成功→跳转首页 |
| 宠物查询 | 搜索宠物→查看详情→返回 |
| 论坛浏览 | 进入论坛→浏览帖子→查看详情 |
| 论坛互动 | 发帖→回复→点赞→删除 |
| 第三方登录 | 微信/QQ登录→授权→进入首页 |

**移动端（APK）**：Python + Appium

| 自动化场景 | 描述 |
|-----------|------|
| 启动→登录 | 冷启动→登录→进入首页 |
| 悬浮窗功能 | 开启悬浮窗→拖拽→点击→关闭 |
| 横竖屏切换 | 切换方向→布局正确→数据保留 |

### 6.4 自动化执行策略

| 阶段 | 触发方式 | 执行范围 | 预期耗时 |
|------|----------|----------|----------|
| 每次提交 | CI/CD自动触发 | 接口自动化全量 | ≤ 10min |
| 每日构建 | 定时触发(每天8:00) | 接口自动化 + E2E冒烟 | ≤ 30min |
| 发版前 | 手动触发 | 全量自动化 | ≤ 1h |
| 回归测试 | Bug修复后触发 | 关联模块接口 + E2E | ≤ 20min |

### 6.5 不建议自动化的测试

- 视觉UI细节（走马观花式页面检查）
- 复杂手势交互（多指操作、复杂拖拽）
- 第三方APP交互（微信/QQ授权页面）
- 兼容性测试（多机型需人工覆盖）
- 性能主观体验（流畅度感受）

## 七、缺陷管理流程

### 7.1 缺陷生命周期

```
新建(New) → 确认(Confirmed) → 修复中(Fixing) → 待验证(Fixed) → 已关闭(Closed)
                                                        ↗
                                                   重新打开(Reopened)
                                                        ↓
                                                  回到 Fixing
```

### 7.2 缺陷优先级定义

| 优先级 | 定义 | 响应时间 | 修复时间 | 示例 |
|--------|------|----------|----------|------|
| **P0-致命** | 核心功能不可用、数据丢失、安全漏洞 | 30分钟 | 当天 | 登录失败、数据库崩溃 |
| **P1-严重** | 主要功能异常、影响核心流程 | 2小时 | 24小时 | 宠物查询无结果、发帖失败 |
| **P2-一般** | 次要功能异常、体验不佳 | 4小时 | 48小时 | 分页异常、图片加载慢 |
| **P3-轻微** | 视觉问题、文案错误、优化建议 | 8小时 | 下Sprint | 错别字、按钮间距不对 |

### 7.3 缺陷报告规范

每条Bug必须包含：

```
【标题】简洁描述问题（模块+操作+现象）
【环境】OS版本、客户端版本、网络环境
【前置条件】触发Bug需要的准备
【复现步骤】1. ... 2. ... 3. ...
【预期结果】正常应该怎样
【实际结果】实际发生了什么
【严重程度】P0/P1/P2/P3
【附件】截图/录屏/日志
```

### 7.4 缺陷统计指标

| 指标 | 计算方式 | 目标 |
|------|----------|------|
| Bug发现率 | 每Sprint新增Bug数 | Sprint间递减 |
| Bug修复率 | 当Sprint修复数/总数 | ≥ 90% |
| Bugreopen率 | 重新打开数/已修复数 | ≤ 10% |
| P0残留数 | 发版时P0未修复数 | = 0 |
| 千行代码Bug率 | Bug数/千行代码 | ≤ 2 |

### 7.5 每日Bug站会

- QA同步昨日新增Bug + 今日测试计划
- 开发同步Bug修复进度 + 阻塞问题
- 产品经理确认Bug严重程度

## 八、回归测试策略

### 8.1 回归触发条件

| 触发条件 | 回归范围 | 执行方式 |
|----------|----------|----------|
| Bug修复 | 修复模块 + 关联模块 | 自动化 + 手工验证 |
| 新功能上线 | 新功能 + 核心主流程 | 全量回归（自动化为主） |
| 接口变更 | 变更接口 + 调用方接口 | 接口自动化 |
| 依赖库升级 | 全量回归 | 自动化全量 |
| 发版前 | 全量回归 | 自动化 + 手工冒烟 |

### 8.2 回归测试集分级

| 级别 | 内容 | 用例数 | 执行频率 | 耗时 |
|------|------|--------|----------|------|
| **L0-冒烟** | 启动、登录、核心功能 | ~20 | 每次构建 | 5min |
| **L1-核心** | 主流程全覆盖 | ~80 | 每日构建 | 20min |
| **L2-功能** | 全功能覆盖 | ~200 | 发版前 | 1h |
| **L3-完整** | 全量用例+兼容性 | ~360 | 大版本发版 | 4h |

### 8.3 回归测试执行流程

```
1. 代码合并 → 触发CI构建
2. 自动执行L0冒烟测试
   ├── 通过 → 继续L1核心回归
   └── 失败 → 阻断，通知开发
3. L1核心回归
   ├── 通过 → 标记"可进入下一阶段"
   └── 失败 → 定位失败用例，关联Bug
4. 发版前执行L2功能回归
5. 大版本执行L3完整回归（含兼容性）
```

### 8.4 回归测试自动化目标

| Sprint | 自动化覆盖率目标 | 重点 |
|--------|-----------------|------|
| Sprint 1-2 | 接口自动化 60% | 用户模块、宠物查询 |
| Sprint 3-4 | 接口自动化 80% | 论坛模块、E2E冒烟 |
| Sprint 5-6 | 接口自动化 95% | 后台管理、全链路E2E |
| Sprint 7-8 | 全面 95%+ | 全量回归自动化 |
